# OCR 読み取り強化 設計書（OCR v6）

> 目的：L1M配送表OCRの読み取り精度・自動救済率・レビュー効率を、**OCR.spaceのみ・1画像1回**の絶対方針を維持したまま最大化する。
> 最終更新: 2026-07-03

---

## 0. 絶対制約（不変）
- OCR.space のみ（Cloud Vision / Gemini / AI fallback 禁止）
- 1画像につきOCR.space送信は1回（複数パターン多重送信禁止）
- 個人情報を console.log / public 保存しない
- 予測値を確定値扱いしない（source/confidence/status 付与）
- ADMIN_APPROVED / MANUAL_FIXED は自動処理で上書き禁止

## 1. 調査結果（自コード＋外部知見）

### OCR.space パラメータ（外部調査）
| 項目 | 結論 |
|---|---|
| OCREngine | 表＋日本語＋**座標(overlay)必須**の当システムでは **Engine2** が最適。Engine3はmarkdown表形式で**単語座標を返さず**、座標ベースパーサーと非互換のため不採用。Engine1は日本語対応だが座標精度が劣る |
| isTable | true（行単位保証・表/伝票向け）→ 採用済み |
| scale | true（内部アップスケールで低解像度に有効）→ 採用済み |
| detectOrientation | true（自動回転）→ 採用済み |

→ **現行パラメータは表OCRとして最適**と確認。伸びしろは「前処理」と「座標パーサーの堅牢化」。

### 自コード調査で判明した弱点
1. **数量列が左→右に整列されておらず**、常温/クーラー/ケース/総数の割当がズレ得る（実害大）
2. 配車Noの誤読（ハイフン消失・l↔1）で**ブロック開始を取りこぼす**
3. 数量語に非数値ノイズが混じると誤割当
4. 前処理が2600px・固定値ベタ書きで本番調整不可
5. 誤読辞書に S→5 / B→8 / | → 1 / ハイフン統一が未収録
6. スキャンPDFがEngine2で失敗（PDF非対応）
7. 画像品質チェックのTODO（端切れ/影/白飛び）未実装

## 3. 実装（要・実画像検証／次段階）

以下は**実際の配送表画像で「今どう崩れるか」を見てから**調整すべき（ブラインドで数値を変えると精度が逆行するリスク）。

- 列境界 `leftBoundary=0.12` / `quantityBoundary=0.76` の最適化（実レイアウト実測）
- y座標クラスタリングの動的tolerance
- 複数行住所の結合ルール（行間しきい値）
- ページ跨ぎの同一帳票判定・重複明細除去
- Engine2の日本語誤読の実傾向に基づく誤読辞書の追記（住所地名は完全一致のみ）

### 検証ループ（推奨手順）
1. 実配送表を `/driver/camera` または管理カメラで取込
2. `/admin/ocr-review/[id]` と `/admin/import-accuracy` で崩れ方・要確認理由TOPを確認
3. 具体的な崩れに対して列境界・tolerance・辞書を微調整（1変更ごとに再検証）
4. 修正履歴（`ocr_correction_patterns`）に安全補正が蓄積され、次回以降自動適用（usageCount≥2）
